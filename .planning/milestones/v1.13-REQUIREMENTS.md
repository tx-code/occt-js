# Requirements: occt-js

**Defined:** 2026-04-22
**Core Value:** Downstream applications can reliably consume the OCCT Wasm runtime and its root API contract without build drift or packaging surprises.

## v1 Requirements

Active milestone: `v1.13 CAM Measurement Integration Sample`

### Demo CAM Workflows

- [x] **DEMO-06**: The browser demo exposes a narrow set of CAM-flavored exact measurement workflows such as `clearance / step depth`, `center-to-center`, and `surface-to-center` by composing the shipped exact surface instead of introducing mesh heuristics or inspection-product behavior.
- [x] **UX-05**: Demo measurement actions and current-result summaries use CAM-oriented naming where the workflow semantics are known, while the browser experience remains a sample-first integration surface rather than a full inspection workspace.

### Boundary Discipline

- [x] **BOUND-02**: CAM workflow semantics stay demo-owned over the existing runtime/package exact surface; any new root/package API work must be justified by a concrete reusable primitive gap rather than by workflow naming, reporting, or session policy.

### Docs & Governance

- [x] **DOCS-08**: Root/package/demo docs and planning truth describe the CAM measurement sample accurately, including the split between reusable exact primitives and demo-owned CAM workflow composition.
- [x] **GOV-06**: Conditional verification and browser coverage fail on drift in the CAM demo workflow while keeping `npm run test:release:root` unchanged as the authoritative root release gate.

## Deferred

- **FEAT-10**: Whole-model CAM analysis such as minimum-radius heatmaps, accessibility / undercut analysis, or deviation-coloring workflows can be reconsidered only in a dedicated future milestone.
- **PROD-01**: Tolerance reports, probing result import, PDF inspection output, or enterprise PMI/CMM flows remain separate product scope and are not implied by this milestone.

## Out of Scope

| Feature | Reason |
|---------|--------|
| New broad exact-runtime APIs for CAM workflow naming alone | The current exact surface already ships the primitives needed for the planned demo-first workflows |
| PMI / GD&T authoring, CMM programming, or enterprise quality-data loops | Those workflows exceed the current `occt-js` runtime/package boundary |
| Probing result import, tolerance management, PDF reporting, or persistent inspection history | Those are app/product concerns above the current sample-first demo |
| Whole-model feature mining such as automatic candidate discovery, minimum-radius heatmaps, or accessibility analysis | These need explicit future scope rather than being smuggled into a narrow demo-first milestone |
| Desktop-first CAM workflow work that changes root publishing assumptions | The root Wasm runtime and browser demo remain the primary maintained surfaces |
