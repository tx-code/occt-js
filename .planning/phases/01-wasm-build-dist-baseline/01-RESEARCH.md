## User Constraints (from CONTEXT.md)

### Locked Decisions

### Build scope
- **D-01:** Treat this as a Windows-first reproducibility phase. Linux/macOS parity, CI rollout, and desktop-specific packaging changes are outside this phase unless they are required to preserve the existing `dist/` contract.
- **D-02:** Keep the current repository structure. The build contract remains `build/wasm/*` for intermediates and `dist/occt-js.js` plus `dist/occt-js.wasm` for distributable runtime outputs.

### Artifact policy
- **D-03:** `dist/occt-js.js` and `dist/occt-js.wasm` remain the canonical runtime artifacts consumed by root tests, local web development, and Tauri bundling.
- **D-04:** `dist/occt-js.d.ts` is a tracked package asset and must be preserved or restored from git; phase work must not assume the Wasm build regenerates it.

### Verification bar
- **D-05:** Phase completion must be proven through the documented clean-Windows flow, prerequisite/error-path coverage, and root runtime preflight behavior before claiming the build path is healthy.
- **D-06:** Root verification is primary in this phase. Demo and desktop surfaces remain downstream consumers of the `dist/` contract, but they are not the main verification target for this build-baseline hardening step.

### Failure diagnostics
- **D-07:** Build failures should continue to retain a predictable log file at `build/wasm-build.log` with enough information to retry or diagnose.
- **D-08:** The low-parallel retry path using `BUILD_JOBS=1` stays part of the expected Windows troubleshooting path rather than being treated as an incidental workaround.

### Claude's Discretion
- Planning can decide the exact split between script hardening, prereq validation, and doc reconciliation as long as the phase goal and artifact contract stay intact.
- Minor developer-experience improvements are allowed when they support the existing build contract, but they must not silently widen phase scope into CI, release automation, or cross-platform redesign.

### Deferred Ideas (OUT OF SCOPE)
- Linux/macOS parity improvements - separate phase once the Windows baseline is stable
- CI verification of clean Wasm rebuilds - future release automation work
- Broader release automation or package publishing orchestration - later governance/release phase

## Architecture Patterns

### Recommended Project Structure

The current repo already separates toolchain, intermediates, distributables, and consumers in the right way for this phase. [VERIFIED: codebase file]

```text
tools/              # Windows setup/build wrappers and shared prereq checker
build/wasm/         # local emsdk plus CMake/Make intermediates
dist/               # canonical distributable runtime contract
test/               # root preflight + runtime tests that load dist artifacts
demo/               # browser dev consumer of local dist/
demo/src-tauri/     # desktop bundle that includes dist/ resources
```

### Pattern 1: Setup -> Preflight -> Build -> Dist Verify

**What:** The Windows path is already layered: setup provisions the toolchain, `check_wasm_prereqs.mjs` validates local markers, `build_wasm_win.bat` configures/builds with `emcmake` and `emmake`, and `build_wasm_win_dist.bat` ensures `dist/occt-js.js` plus `dist/occt-js.wasm` exist before declaring success. [VERIFIED: codebase file][CITED: https://emscripten.org/docs/compiling/Building-Projects.html]

**When to use:** Use this as the only clean-Windows maintainer flow for Phase 01 planning and validation. [VERIFIED: codebase file]

**Example:** [VERIFIED: codebase file]

```bat
node tools\check_wasm_prereqs.mjs windows dist-types || goto :error_prereqs
call build\wasm\emsdk\emsdk_env.bat >> "%LOG_FILE%" 2>&1 || goto :error_env
call emcmake cmake -S . -B build\wasm -G "MinGW Makefiles" -DCMAKE_MAKE_PROGRAM=mingw32-make -DCMAKE_BUILD_TYPE=%BUILD_TYPE% >> "%LOG_FILE%" 2>&1 || goto :error_configure
call emmake mingw32-make -C build\wasm -j%BUILD_JOBS% >> "%LOG_FILE%" 2>&1 || goto :error_build
```

### Pattern 2: Dist-First Runtime Verification

**What:** Root tests load `../dist/occt-js.js` and fail with explicit guidance when it is missing instead of rebuilding Wasm implicitly. [VERIFIED: codebase file]

**When to use:** Use this pattern for all Phase 01 verification that is supposed to validate the published/runtime contract rather than the build system itself. [VERIFIED: codebase file]

**Example:** [VERIFIED: codebase file]

```js
export function createOcctRequireError(factoryPath) {
  return new Error(
```

### Pattern 3: Downstream Surfaces Consume `dist/`, Not `build/wasm`

**What:** The root package publishes `dist/occt-js.js`, `dist/occt-js.wasm`, and `dist/occt-js.d.ts`; demo dev mode resolves `../../../dist/`; Tauri bundles `../../dist/occt-js.js` and `../../dist/occt-js.wasm` as resources. [VERIFIED: codebase file]

**When to use:** Keep every Phase 01 change anchored to the `dist/` contract rather than letting any consumer depend on generator-specific intermediate paths. [VERIFIED: codebase file]

**Example:** [VERIFIED: codebase file]

```json
{
  "main": "dist/occt-js.js",
  "types": "dist/occt-js.d.ts",
```

### Anti-Patterns to Avoid

- **Repo-root or shared `emsdk` assumptions:** Repo instructions explicitly reject that path on Windows; Phase 01 should preserve `build/wasm/emsdk` as the only documented clean-worktree location. [VERIFIED: codebase file]
- **Treating `dist/occt-js.d.ts` as generated output:** It is tracked in git and the prereq checker already tells maintainers to restore it with `git restore` if missing. [VERIFIED: codebase file][VERIFIED: shell command]
- **Implicit Wasm builds inside root tests:** `package.json` and `load_occt_factory.mjs` already separate build and runtime verification, and the plan should keep that separation. [VERIFIED: codebase file]
- **Making demo or Tauri a primary release gate in this phase:** Repo guidance says the desktop app is additive and the root package publishing flow must remain independent. [VERIFIED: codebase file]
- **Leaving artifact-origin ambiguity unresolved:** The plan should not assume `build/wasm` and `dist/` can remain dual sources of truth indefinitely. [VERIFIED: codebase file]

## Common Pitfalls

### Pitfall 1: Prerequisite Coverage Stops at File Markers

**What goes wrong:** Missing `occt/src/Standard`, missing `build/wasm/emsdk/emsdk_env.bat`, and missing `dist/occt-js.d.ts` already fail with actionable messages, but missing Git during setup or missing CMake during configure are not surfaced by the same shared checker. [VERIFIED: codebase file]

**Why it happens:** `tools/check_wasm_prereqs.mjs` only validates file markers, while `tools/setup_emscripten_win.bat` calls `git clone` directly and `tools/build_wasm_win.bat` reaches generic `:error_configure` / `:error_build` branches only after invoking external commands. [VERIFIED: codebase file]

**How to avoid:** Decide in planning whether Phase 01 should add explicit Git/CMake detection or whether the docs will treat those as hard prerequisites and rely on retained logs for diagnosis. [VERIFIED: codebase file]

**Warning signs:** `Setup failed with error %errorlevel%.` from setup, or `Configure failed.` / `Build failed.` with details only in `build/wasm-build.log`. [VERIFIED: codebase file]

### Pitfall 2: Two Competing Artifact-Origin Assumptions

**What goes wrong:** `CMakeLists.txt` directs runtime outputs to `dist/`, but the Windows dist wrapper still copies `build\\wasm\\occt-js.js` and `build\\wasm\\occt-js.wasm` into `dist/`. [VERIFIED: codebase file][CITED: https://cmake.org/cmake/help/latest/prop_tgt/RUNTIME_OUTPUT_DIRECTORY.html]

**Why it happens:** The CMake layer and the Windows wrapper encode different ideas about where the authoritative generated files originate before distribution. [VERIFIED: codebase file]

**How to avoid:** Make the plan explicitly reconcile and test the authoritative output path on a clean Windows build before any docs or additional checks are updated. [VERIFIED: codebase file]

**Warning signs:** Any successful build path that leaves uncertainty about whether `dist/` was generated directly or only became correct because of a copy step. [VERIFIED: codebase file]

### Pitfall 3: Full Root Verification Requires Built `dist/` Artifacts

**What goes wrong:** The fast `node --test` preflight checks pass without a full build, but the full `npm test` chain will fail until `dist/occt-js.js` exists because runtime scripts call `loadOcctFactory()`. [VERIFIED: codebase file][VERIFIED: shell command]

**Why it happens:** The repo intentionally separates prerequisite/loader messaging tests from runtime contract tests that require actual built artifacts. [VERIFIED: codebase file]

**How to avoid:** Keep a clear quick-run command for preflight coverage and a separate build-plus-full-suite gate for phase completion. [VERIFIED: codebase file]

**Warning signs:** `Missing .../dist/occt-js.js. Build the Wasm artifacts first with "npm run build:wasm:win"...` from `createOcctRequireError()`. [VERIFIED: codebase file]

### Pitfall 4: Tracked Type Definitions Can Be Deleted by Cleanup Habits

**What goes wrong:** Clean/rebuild work can remove `dist/occt-js.d.ts`, but the Wasm build will not recreate it. [VERIFIED: codebase file][VERIFIED: shell command]

**Why it happens:** The root package treats the type definition file as a tracked publish artifact, not generated build output. [VERIFIED: codebase file][VERIFIED: shell command]

**How to avoid:** Preserve the current prereq check and docs language that restore the file from git instead of rebuilding it. [VERIFIED: codebase file]

**Warning signs:** `node tools/check_wasm_prereqs.mjs dist-types` fails and instructs `git restore --source=HEAD -- dist/occt-js.d.ts`. [VERIFIED: codebase file]
