## Workspace topology

| Option | Description | Selected |
|--------|-------------|----------|
| Actor-scoped workspace in demo store | Keep workpiece and tool as separate retained actors with ids, roles, pose, and lifecycle state | ✓ |
| Synthetic merged assembly result | Combine workpiece and tool into one heavier scene/exact model before the demo interacts with them | |
| Singular model plus ad-hoc tool overlay | Keep the current single-model store and stash the tool outside the authoritative workspace contract | |

**User's choice:** Actor-scoped workspace in demo store
**Notes:** The workpiece-plus-tool workflow needs simultaneous retained state, clean ownership, and explicit pose semantics. A synthetic merge would obscure lifecycle and movement rules too early, while an ad-hoc overlay would reintroduce viewer-only drift.

---

## Tool pose ownership

| Option | Description | Selected |
|--------|-------------|----------|
| Store-owned actor pose shared by render and exact flows | Tool movement updates one actor transform consumed by both Babylon rendering and exact-ref / measurement logic | ✓ |
| Babylon-only transform | Move only the rendered mesh and rebuild exact refs later from viewer-local state | |
| Reopen exact model after every movement step | Treat each tool movement as a new exact-open lifecycle event | |

**User's choice:** Store-owned actor pose shared by render and exact flows
**Notes:** The user explicitly asked for a movable tool that remains measurable. That only works if render pose and exact pose stay in lockstep. Reopening exact state on every move would be too heavy and would make lifecycle semantics brittle.

## Cross-model pairwise seam

| Option | Description | Selected |
|--------|-------------|----------|
| Add additive root cross-model pairwise entrypoints and dispatch from `occt-core` | Keep current same-model methods intact while allowing refs from different `exactModelId` values | ✓ |
| Mutate existing root method signatures in place | Change the current root contract so all pairwise calls must pass two exact-model ids | |
| Avoid kernel work and fake it in the demo | Approximate measurements or force one exact assembly flow instead of adding explicit cross-model support | |

**User's choice:** Add additive root cross-model pairwise entrypoints and dispatch from `occt-core`
**Notes:** Current pairwise wrappers already prove the kernel semantics, but they hard-require one shared `exactModelId`. The workpiece-plus-tool workflow makes that gap concrete enough to justify a narrow additive extension without rewriting the existing contract in place.

---

## Actor-scoped selection contract

| Option | Description | Selected |
|--------|-------------|----------|
| Resolve actor-scoped exact refs at selection time | Each selection record carries actor identity plus a measurement-ready exact ref composed from occurrence transform and actor pose | ✓ |
| Keep UI-only selection summaries until a command runs | Delay exact-ref resolution until measurement execution time | |
| Use Babylon ids as the bridge contract | Let mesh ids or instance ids leak into downstream measurement actions | |

**User's choice:** Resolve actor-scoped exact refs at selection time
**Notes:** This keeps Phase 34 focused on measurement actions instead of rebuilding topology context. It also preserves the repo-wide rule that Babylon ids stay UI-local rather than becoming geometry contracts.

## the agent's Discretion

- Whether tool movement ships first as numeric transform controls or constrained nudge actions, as long as the pose contract is explicit and testable
- Exact additive naming for root cross-model pairwise entrypoints, as long as existing same-model names remain source-compatible
