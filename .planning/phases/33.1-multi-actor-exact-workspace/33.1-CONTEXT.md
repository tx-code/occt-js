# Phase 33.1: Multi-Actor Exact Workspace - Context

**Gathered:** 2026-04-21
**Status:** Inserted and awaiting planning

<domain>
## Why This Phase Was Inserted

The original `v1.10` plan assumed a single retained exact session in the demo. That was sufficient for one loaded model plus selection-to-measure bridging, but it is not sufficient for the newly confirmed target workflow: import a workpiece, generate a tool, move the tool, and measure between them.

This turns two earlier assumptions into active blockers:

- one global demo `exactSession` is no longer enough
- same-`exactModelId` pairwise exact measurement is no longer enough

Phase `33.1` exists to resolve those blockers before the measurement MVP is implemented.

</domain>

<decisions>
## Implementation Decisions

- The demo must support at least two actor families in one workspace: an imported workpiece and a generated tool.
- Each actor must own its own retained exact session, rendered model identity, and pose state.
- Tool movement must update the transform used for exact refs and pairwise exact measurement, not only the Babylon scene node transform.
- Babylon entity ids remain UI-local; selection and measurement contracts must stay actor-scoped and occurrence-safe.
- Additive cross-model exact pairwise support is now justified by a concrete demo requirement, but broader unrelated exact API growth remains out of scope.

</decisions>

<specifics>
## Specific Planning Targets

- Define a multi-actor workspace store contract that can host at least a workpiece actor and a tool actor simultaneously.
- Decide whether the generated tool remains an independent exact model with actor-scoped transforms or is merged into a heavier shared exact assembly flow. The independent-actor path is the current default.
- Plan the narrowest additive runtime or package change needed to support pairwise exact measurements across different retained exact models.
- Re-derive the old `33-02` selection bridge around actor ids, actor pose, and actor-scoped exact refs rather than one global demo exact session.

</specifics>

<deferred>
## Deferred

- Toolpath simulation, collision detection, gouge checks, or CAM verification beyond exact measurement
- Full gizmo-rich interaction design or persistent measurement sessions
- Broad assembly authoring beyond the workpiece-plus-tool workspace needed for the measurement MVP

</deferred>
